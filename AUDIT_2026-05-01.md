## 1. THE BIG PROBLEM

The app shows fake numbers across many pages without telling the user. A beginner cannot tell what is real and what is decorative. The app's stated principle is "beginners-first, every recommendation explained." The fake-data problem violates that principle directly.

In addition to fake data, the app has accumulated:
- Functional bugs that have been documented but not fixed (auth bypass, broken auto-refresh, mode initialisation issues)
- Multiple competing UI patterns for the same task (5 ticker pickers, 3 mode cockpits)
- UI issues that fail accessibility standards (touch targets, font size, no focus rings, no screen reader labels)
- Security issues (4 unauthenticated routes)

**BUG-20 (hardcoded 1.5% std) — RESOLVED.** `/api/var` now returns computed `portfolio_std` from actual positions (0.5476 in this run, not 0.015). The audit's prior "hardcoded 1.5%" claim is outdated.

Worth a deeper audit of the std calculation — but the architecture is real, not hardcoded.

## 8. BACKEND MATH + LOGIC — VERIFIED BY CODE READING

These items in Section 13 of prior versions of this audit are now VERIFIED by reading the relevant code paths:

### 8.1 Indicator math — VERIFIED CORRECT
- `rma()` (line 259): Wilder's RMA. alpha=1/length, SMA seed, recursive. Correct definition. ✓
- `ema_tv()` (line 273): TradingView-style EMA. alpha=2/(span+1), SMA seed. Matches Pine Script. ✓
- `get_rsi()` (line 480): `100 - 100/(1 + rma(gain)/rma(loss))`. Wilder RSI. Correct. ✓
- `detect_rsi_divergence()` (line 487): proper pivot-based scan with bullish/bearish/hidden divergence. Pivot length 3, lookback 100. Correct logic. ✓
- MACD (line 705): line = EMA(12) − EMA(26), signal = EMA(line, 9), hist = line − signal. Correct. ✓
- Bollinger Bands (line 709): 20-period SMA ± 2 × std. Uses `ddof=0` (population std). TradingView uses ddof=1 (sample std). Tiny difference (sqrt(20/19) ≈ 1.026×). Acceptable but not exactly TV-compatible. **Minor discrepancy noted, not a bug.**
- ATR (line 717): TR = max(high-low, |high-prev close|, |low-prev close|). Then `rma(tr, 14)` smoothed by 100-bar rolling mean for stop sizing. The double-smoothing (Wilder + 100-bar mean) is unusual but documented as intentional ("raw ATR14 too noisy"). ✓
- Supertrend (line 734): HL2 ± 3.0 × ATR(10), with proper final-band tracking and direction logic. Correct standard implementation. ✓
- EMA trend classification (line 760): 5-tier (STRONG BULL / BULL / MIXED / BEAR / STRONG BEAR) based on price/e20/e50/e200 ordering. Sensible. ✓

**Verdict on indicator math:** the math is correct. No silent computation bugs found.

### 8.2 Watch / alert firing logic — VERIFIED WORKING
`run_watch_job()` (line 2825):
- Runs every 60s
- Per-ticker timing controlled by `ALERT_INTERVALS` (5m → 300s, 15m → 600s, 30m → 900s, 1h → 1800s, 4h → 3600s, 1d → 14400s). Sensible.
- Fetches data, calculates indicators, runs `pre_screen`
- If `pre_screen` says "call_claude", runs `get_watch_signal`
- Compares to previous signal — fires only if NEW actionable signal (was HOLD, now BUY/SELL)
- Or if counter-trade detected
- Calls `fire_alert` with channels (sms, telegram, etc.)
- On successful delivery, REMOVES the watch (one-shot per signal change)
- On failed delivery, keeps last_signal so it retries next tick
- Live narrative update even when no new signal — keeps watch panel current

**Verdict:** logic is correct and well thought out. ✓
